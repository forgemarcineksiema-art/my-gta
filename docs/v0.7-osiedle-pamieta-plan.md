# v0.7 Plan: Przypal + World Memory + NPC Reactions Lite

## Summary

v0.7 should make the current block feel like it notices the player. This is not a big AI update and not a new mission chain. It is the first "osiedle pamieta" pass: Przypal rises from visible actions, recent events are remembered for a short time, and simple NPC reaction emitters comment or apply light pressure.

The goal is to create the first system loop:

```txt
player causes chaos
-> WorldEventLedger records it
-> PrzypalSystem raises social heat
-> NPCReactionSystem chooses a small reaction
-> HUD/camera/feedback communicate pressure
-> event decays or is referenced later
```

Hard architecture rule:

```txt
WorldEventLedger is memory, not a heat source.
Przypal rises only from WorldEventAddResult returned by ledger.add().
Active events must never be scanned every frame to add heat again.
```

## Scope

Implement only a small set of event types first:

- `PublicNuisance`: bumping props, repeated sprint chaos, loud public action.
- `PropertyDamage`: trash/bin/parking prop hit or strong collision.
- `ChaseSeen`: chase pressure near public areas.
- `ShopTrouble`: chaos near shop/mission marker.

Implement only a few reaction sources first:

- babka z okna: alarm line + HUD/feedback pulse;
- zul/gossip marker: comments on recent event;
- shopkeeper/mission NPC: nervous or annoyed line near shop;
- local chaser/patrol hint: appears only as pressure feedback, not full police AI.

## Non-Goals

- No full faction reputation.
- No persistent save-game world memory.
- No economy.
- No new district.
- No complex NPC navigation.
- No combat implementation.
- No large dialogue tree.
- No procedural quest generator.

## Core Interfaces

### `WorldEventLedger`

Stores recent events.

Event fields:

- type;
- position;
- intensity;
- time remaining;
- source tag;
- repeat count or stack count.

Required behavior:

- add event;
- merge nearby repeated events when useful;
- expire old events;
- query recent events by type/radius/intensity.
- return a one-shot `WorldEventAddResult` pulse.
- dampen repeated merges: first event `1.0`, second stack `0.45`, stack `>= 3` `0.25`.
- cap stack count at `5`; further repeats refresh memory but do not emit heat.

### `PrzypalSystem`

Owns current social heat.

Suggested state:

- current value `0..100`;
- target value for smoothing;
- decay rate;
- band: calm, noticed, hot, chase-risk, meltdown;
- recent contributors for debug.

Required behavior:

- accept event contribution only through `onWorldEvent(result)`;
- decay over time;
- expose normalized intensity to HUD, `GameFeedback`, and `CameraRig`;
- never directly steal player control.

Defaults:

- `PublicNuisance +8`
- `PropertyDamage +16`
- `ChaseSeen +12`
- `ShopTrouble +18`
- `decayDelay = 1.5s`
- `decay = 3.0/s`
- bands: Calm, Noticed, Hot, ChaseRisk, Meltdown.

### `NpcReactionSystem`

Consumes event ledger + Przypal + player position and emits reaction intents.

Reaction intent examples:

- subtitle line;
- marker pulse;
- heat reason/debug metadata, without adding heat outside event pulses;
- comedy feedback event;
- future hook for NPC behavior.

Required behavior:

- per-reaction cooldown;
- global anti-spam cooldown;
- priority rules so chase/fail UI wins over gossip;
- simple deterministic tests.
- multiple line variants per source.
- no immediate repeat of the same source line.
- mission-critical UI can downgrade high-priority reactions to HUD-only and suppress low-priority gossip.

### `WorldInteraction`

Small glue layer for placing reaction markers in the current primitive map:

- shop reaction zone;
- window/babka alarm zone;
- gossip/zul marker;
- parking/trash event zone.

## App Layer Integration

`GameApp.cpp` should stay presentation-focused:

- convert collisions/mission/chase moments into world events;
- draw compact Przypal HUD band;
- play reaction subtitles through existing `DialogueSystem` or a small reaction subtitle path;
- pass Przypal intensity into `GameFeedback` and `CameraRig`;
- show debug overlay: Przypal value, active events, last reaction, cooldowns.
- keep reaction decision rules inside `NpcReactionSystem`, not in `GameApp.cpp`.
- use emitter cooldowns for `ChaseSeen`, `PropertyDamage`, horn/noise, and shop events.

## Tests

Core tests should be added before implementation:

- event ledger stores and expires events;
- repeated nearby events stack or merge without infinite growth;
- active event memory does not add heat without a new add result;
- Przypal rises from events and decays to calm;
- Przypal caps at 100;
- band thresholds and HUD labels are stable;
- emitter cooldowns block `ChaseSeen` spam;
- damage below threshold does not emit a property-damage event;
- NPC reaction fires once, then respects cooldown;
- high-priority chase/fail state suppresses low-priority gossip spam;
- reaction variants do not repeat the same line twice;
- Przypal intensity affects feedback values but does not alter player movement direction.

## Manual Playtest

Run after implementation:

- create chaos near shop and confirm shop/NPC reaction;
- create repeated parking/trash chaos and confirm Przypal rises;
- wait without chaos and confirm Przypal cools down;
- trigger chase near public area and confirm stronger pressure;
- verify HUD is readable at 720p;
- verify camera shake/FOV never makes movement direction confusing;
- verify NPC lines do not spam;
- verify current mission still works.

## Acceptance Criteria

- Player understands why Przypal rose.
- At least three different actions can produce visible social heat.
- At least three reaction sources can comment or pulse without new mission content.
- The world feels slightly more alive after one minute of free roam.
- Movement/camera complaints do not get worse.

## Next After v0.7

If v0.7 works, v0.8 can deepen one of these:

- vehicle/chase identity tied to Przypal;
- one mission rewritten to use world memory;
- first simple faction/respect variable;
- indoor/staircase camera and interaction polish.
